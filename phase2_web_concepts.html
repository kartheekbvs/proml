Web, HTTP & ASGI Servers

Master how Machine Learning models travel across the internet. From bytes to predictions.

The Mission: Learn how to bridge the gap between a trained Python model and a user's browser. We aren't just building scripts anymore; we are building services.

Focus: Moving from PHP's Request/Response cycle to modern Python Asynchronous servers.

Prerequisites: Completion of Phase 1 (Python Basics).

Est. Time: 90 Minutes.

01. Why Web Concepts?

The Great Trap: Many beginners think Machine Learning is only about math and Jupyter notebooks. But a model that sits on your hard drive is useless. To make it valuable, it must live on a Server.

In PHP, you were used to Apache or Nginx handling everything. In Python backend development, you often configure the server layer (Uvicorn/Gunicorn). Understanding the web is the difference between a prototype and a product.

[ User Browser ] <---> [ The Cloud ] <---> [ Your ML API ] | | | "Categorize this" HTTP NumPy/PyTorch

02. Core Concept: Everything is a Request

What is this? At its simplest, the web is a conversation. A client asks (Request) and a server answers (Response). Whether you are loading a cat photo or asking an AI to write a poem, the mechanism is the same.

The rules of this conversation are called HTTP (HyperText Transfer Protocol). Since ML models often take time to "think," we need modern servers that can talk to many people while the model works in the background (Async).

03. Mental Model: The Restaurant Waiter

To understand the difference between old Python servers (WSGI) and new ones (ASGI), use the Waiter Analogy:

Sync (WSGI/PHP-FPM) The waiter takes your order, then stands in the kitchen waiting for the chef to cook. He cannot serve anyone else until your plate is ready.
Async (ASGI/FastAPI) The waiter takes your order, puts it in the kitchen, and mid-cook, moves to serve other tables. He only returns to you when your food is ready.

Because Machine Learning models are like "slow chefs," the Async model is essential for high-performance AI apps.

04. First Example: The "Mock" ML API

Instead of a full model, let's look at the "Web Plumbing" required to handle a request. We will use SimpleHTTP to demonstrate raw responses before moving to frameworks.

api_core.py
import json
from http.server import BaseHTTPRequestHandler, HTTPServer

# This class handles individual conversations (requests)
class MLMockHandler(BaseHTTPRequestHandler):
    
    # This runs when a user sends a POST request
    def do_POST(self):
        # 1. Read how many bytes the user sent
        content_length = int(self.headers['Content-Length'])
        
        # 2. Get the actual data (JSON)
        post_data = self.rfile.read(content_length)
        input_data = json.loads(post_data) # Convert string to Python Dict

        # 3. Simulate a "Model Prediction"
        prediction = f"Likely {input_data['text'].upper()}!!"
        
        # 4. Prepare the Response
        response = {
            "status": "success",
            "prediction": prediction
        }

        # 5. Send Headers (The metadata)
        self.send_response(200) # 200 means OK
        self.send_header('Content-type', 'application/json')
        self.end_headers()

        # 6. Send the Body (The actual data)
        self.wfile.write(json.dumps(response).encode('utf-8'))

# Run the server on port 8000
if __name__ == "__main__":
    server_address = ('', 8000)
    httpd = HTTPServer(server_address, MLMockHandler)
    print("🚀 API Bouncer is standing at port 8000...")
    httpd.serve_forever()

05. Line-by-Line Code Walkthrough

1
BaseHTTPRequestHandler: This is a skeleton provided by Python's standard library. It handles the low-level socket connections so you don't have to.
2
self.headers['Content-Length']: In HTTP, the server needs to know exactly how many characters to read. If you miss this, the server will wait forever!
3
self.wfile.write(...): This sends the data back to the user's browser. We must .encode('utf-8') because the internet doesn't send "text," it sends "bytes."

SUB-TOPIC 2.1

How the Web Works (Under the Hood)

When you type google.com, five things happen in milliseconds:

  1. DNS Lookup: Your computer asks a phonebook (DNS) for the IP address of the server (e.g., 142.250.190.46).
  2. TCP Connection: A "handshake" occurs to ensure both sides are ready to talk.
  3. HTTP Request: You send a message like GET /index.html HTTP/1.1.
  4. Processing: The server runs your Python/ML code.
  5. HTTP Response: The server sends back 200 OK with the data.

Client vs Server Role

The Client (Consumer) Asks for things. Handled by Browsers, Mobile Apps, or other Servers.
The Server (Provider) Has the data/logic. Handled by Python, FastAPI, Flask, etc.
SUB-TOPIC 2.2

REST APIs & CRUD

REST (Representational State Transfer): A fancy name for a simple rule: Use HTTP methods as "verbs" and URLs as "nouns."

The HTTP Verbs (The "V" in CRUD)

VERB | CRUD ACTION | USE CASE -------------------------------------------------- GET | Read | "Get me user #5" POST | Create | "Create a new user" PUT | Update | "Replace user #5's profile" PATCH | Update | "Change just the user's email" DELETE | Delete | "Delete user #5"

HTTP Status Codes: The Server's Mood

  • ✅ 200 OK: Everything went perfect.
  • 🆕 201 Created: Successfully made a new item.
  • ❌ 400 Bad Request: You (client) sent garbage.
  • 🔒 401 Unauthorized: Where is your ID/Token?
  • 🔎 404 Not Found: I have no idea what page that is.
  • ⚠️ 500 Server Error: My Python code crashed/is broken.

JSON: The Language of APIs

JSON (JavaScript Object Notation) is Text. Python cannot "do math" on text. We must convert it to a Dictionary first.

SUB-TOPIC 2.3

Servers: WSGI vs ASGI

In the "Old Days" (Flask, Django), Python used WSGI. It followed the "One request per worker" rule. If you had 4 workers, you could only handle 4 people at a time.

Modern ML Servers (ASGI)

ASGI stands for Asynchronous Server Gateway Interface. It allows a single server process to handle thousands of concurrent connections. This is what FastAPI uses.

The Modern Command
# Uvicorn is the "Engine" that runs your code
uvicorn main:app --reload --port 8000

Gunicorn vs Uvicorn: In production, we often use Gunicorn as the manager, and it creates several Uvicorn workers.
Hierarchy: Gunicorn (Boss) -> Uvicorn Workers (Staff) -> Your Code (Kitchen).

N-3. Mini Project: The "Translation" Status API

Build an API that accepts text and returns its "Sentiment" (Mocked) and its length.

sentiment_api.py
import json
from http.server import HTTPServer, BaseHTTPRequestHandler

# In PHP, this would be a .php file with $_POST calls
class SentimentAPI(BaseHTTPRequestHandler):
    
    def do_POST(self):
        # Read the incoming data stream
        content_length = int(self.headers['Content-Length'])
        body = self.rfile.read(content_length).decode('utf-8')
        
        try:
            # Parse the JSON
            payload = json.loads(body)
            text = payload.get("text", "")
            
            # Logic: If it contains 'good' -> Positive
            sentiment = "POSITIVE" if "good" in text.lower() else "NEUTRAL"
            
            # Prepare result
            result = {
                "text": text,
                "length": len(text),
                "sentiment": sentiment,
                "processed_by": "Python-Phase-2-Server"
            }
            
            # Send Success Response
            self.send_response(200)
            self.send_header('Content-Type', 'application/json')
            self.end_headers()
            self.wfile.write(json.dumps(result).encode('utf-8'))
            
        except Exception as e:
            # Send Error Response
            self.send_response(400) # Bad Request
            self.end_headers()
            self.wfile.write(json.dumps({"error": str(e)}).encode())

if __name__ == "__main__":
    httpd = HTTPServer(('localhost', 8080), SentimentAPI)
    print("🧠 Sentiment API is live at http://localhost:8080")
    httpd.serve_forever()

N-2. Quiz & Interview Questions

POST is for creating something new (like a new blog post). PUT is for replacing/updating an existing resource at a specific ID.

ML models are slow. In a synchronous (WSGI) server, the entire server "freezes" while a model is calculating (inference). ASGI allows the server to keep accepting new requests while the model works.

It means the server understood your JSON format, but the data inside was wrong (e.g., expected a number but got "ABCD"). FastAPI gives this error a lot!
